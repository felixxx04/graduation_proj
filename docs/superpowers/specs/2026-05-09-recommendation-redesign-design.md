# 医疗推荐系统架构重构设计

## 背景

基于 DeepSeek 对 204 种疾病的推荐结果验证（适用率 61.1%，部分适用率 37.5%，不适用率 1.4%），识别出两个核心问题：

1. **推荐不出药**：部分疾病（病毒感染、上呼吸道感染、肺纤维化等）无法产生任何推荐结果
2. **推荐错药**：部分疾病的推荐药物类别错误（如抗生素治病毒、PPI 治胆囊病等）

根因是**中文疾病名到英文药品适应症之间的语义鸿沟**——疾病映射器将非标准疾病名路由到错误的代理词，导致药品匹配失败。

同时，系统角色从"医生输入疾病"改为"患者输入口语化症状 → 系统推荐 → 医生审核确认"。

## 方案选择

三个候选方案对比：

| 方案 | 思路 | 风险 | 周期 | 效果 |
|------|------|------|------|------|
| A: 增强匹配层 | 扩充同义词库 + 反向索引 + fallback 链 | 低 | 1-2周 | 改善 40-60% |
| B: 临床知识图谱 | ICD-10 + ATC 分类 + 确定性路由网络 | 中 | 3-4周 | 解决 80-90% |
| C: 两阶段 ML 重排 | 广召回 + 医学 BERT 精排 | 中高 | 4-6周 | 解决 70-85% |

**选择方案 B（临床知识图谱）+ 方案 A 的部分增强，形成混合方案。**

- 不选 C：医疗推荐需要确定性和可解释性，ML 黑盒不可接受
- 不选纯 A：21 条硬编码规则已不可持续，需架构层面根治
- 医生审核环节的加入使得可以适度放宽安全过滤，方案 B 恰好受益于此

---

## 新架构总览

在现有三层架构（SafetyFilter → RuleMarker → DeepFM）之上插入**临床知识路由层**，并增加**医生审核反馈闭环**。

```
患者输入（口语化疾病/症状描述）
       ↓
┌──────────────────────────────────────────────┐
│  新增 · 第零层：临床知识路由层                  │
│                                               │
│  L1 标准化：口语 → 医学术语                     │
│  L2 归类：疾病 → 身体系统 + 病因分类             │
│  L3 适应症映射：系统+病因 → 治疗类别             │
│  L4 药品召回：治疗类别 → 具体药品列表              │
└──────────────────────────────────────────────┘
       ↓
┌──────────────────────────────────────────────┐
│  调整 · 第一层：安全标记层                      │
│  绝对禁忌/过敏 → 硬排除（不变）                  │
│  其他 → 标记待审核，不再拦截                     │
└──────────────────────────────────────────────┘
       ↓
┌──────────────────────────────────────────────┐
│  不变 · 第二层：DeepFM 排序                     │
│  在知识路由限定的候选集内进行 ML 评分             │
└──────────────────────────────────────────────┘
       ↓
┌──────────────────────────────────────────────┐
│  新增 · 医生审核环节                           │
│  ✅确认 / ✏️修改 / ❌拒绝                       │
└──────────────────────────────────────────────┘
       ↓
┌──────────────────────────────────────────────┐
│  新增 · 审核反馈闭环                           │
│  医生决策 → review_log → 定期分析 → 优化路由权重  │
└──────────────────────────────────────────────┘
```

## 核心组件一：临床知识路由层

### 目的

解决"推荐错药"问题。通过四级确定性路由，确保每个疾病对接到正确的药品类别。

### 四级路由结构

| 层级 | 输入 | 输出 | 示例 |
|------|------|------|------|
| L1 标准化 | 患者口语 | 标准医学术语 | "拉肚子" → 腹泻(diarrhea) |
| L2 归类 | 标准术语 | 身体系统 + 病因 | 腹泻 → 消化系统·感染性 |
| L3 适应症映射 | 系统+病因 | ATC 治疗类别 | 消化系统·感染性 → 肠道抗感染药 |
| L4 药品召回 | ATC 类别 | 具体药品列表 | 肠道抗感染药 → Rifaximin, Ciprofloxacin... |

### 数据模型

```python
# L1: 口语标准化表 (colloquial → standard term)
COLLOQUIAL_TO_STANDARD = {
    "拉肚子": ["diarrhea"],
    "嗓子发炎": ["pharyngitis", "upper respiratory infection"],
    "感冒": ["common cold", "upper respiratory infection"],
    "发烧": ["fever"],
    "反酸烧心": ["gastroesophageal reflux disease"],
}

# L2: 疾病归类表 (standard_term → body_system + etiology)
DISEASE_CATEGORIES = {
    "diarrhea": ("gastrointestinal", "infectious"),
    "upper respiratory infection": ("respiratory", "viral"),
    "pneumonia": ("respiratory", "bacterial"),
    "hypertension": ("cardiovascular", "chronic"),
    # ...
}

# L3: 适应症映射表 (body_system + etiology → ATC drug class)
INDICATION_TO_ATC = {
    ("gastrointestinal", "infectious"): ["A07A", "J01X"],  # 肠道抗感染药
    ("respiratory", "viral"): ["R05", "N02B"],              # 镇咳药 + 解热镇痛
    ("respiratory", "bacterial"): ["J01"],                   # 系统性抗生素
    ("cardiovascular", "chronic"): ["C09", "C07", "C08"],   # ACE/ARB + BB + CCB
    # ...
}
```

### 知识路由数据来源

- **ICD-10**：疾病分类标准（身体系统 + 病因归类）
- **ATC Classification**：药品治疗类别标准（WHO ATC/DDD Index）
- **现有 1815 药品 pipeline_data**：药品名 → drug_class_en → 适应症

### 路由特性

- **确定性**：同一种疾病每次路由结果一致，无 ML 随机性
- **可解释性**：每个推荐都可以回溯到路由路径（"推荐 Rifaximin 是因为腹泻 → 肠道感染 → 肠道抗感染药 → Rifaximin 是该类代表药"）
- **可维护性**：路由表是纯数据文件（JSON），发现错误只需修改表项，不需要改代码

## 核心组件二：患者口语增强映射

### 目的

解决"推荐不出药"问题。处理患者非标准口语化表达，确保所有输入都能路由到药品。

### 三层 Fallback 策略

```
Layer 1: 精确匹配
  输入匹配 COLLOQUIAL_TO_STANDARD 中的完整短语
  例："感冒了" → ["common cold", "upper respiratory infection"]
  confidence: "high"

Layer 2: 关键词匹配
  提取输入中的医学关键词，匹配标准术语
  例："喉咙痛" → 关键词 "喉咙" + "痛" → "pharyngitis"
  confidence: "medium"

Layer 3: 症状组合推理（新增）
  多症状组合推理为最可能的疾病类别
  例："头痛 + 发热 + 肌肉酸痛" → 流感类 → 解热镇痛 + 对症
  confidence: "low"
```

### 症状组合推理表（初始规则，可扩展）

| 症状组合 | 推断疾病类别 | 推荐药类 |
|----------|-------------|---------|
| 头痛 + 发热 + 肌肉酸痛 | 流感样疾病 | 解热镇痛药 |
| 尿频 + 尿痛 + 下腹痛 | 尿路感染 | 抗生素 |
| 反酸 + 烧心 + 饭后加重 | GERD | PPI/H2RA |
| 鼻塞 + 流涕 + 打喷嚏 | 过敏性鼻炎 | 抗组胺药 |
| 咳嗽 + 咳痰 + 发热 | 下呼吸道感染 | 抗生素 ± 镇咳祛痰 |
| 关节痛 + 晨僵 + 对称性 | 类风湿关节炎 | NSAIDs ± DMARDs |

### 降级兜底

当三层 Fallback 全部失败时：
- 匹配症状关键词到最接近的药品类别
- 标记 `confidence="low"` + `requires_doctor_review=true`
- 在前端强制显示"必须经医生审核"

## 核心组件三：安全策略从"硬排除"到"标记审核"

### 调整原则

医生审核环节提供了安全网，系统可以从"过度拦截"转为"充分标记"。

### 具体调整

| 场景 | 当前行为 | 新行为 | 理由 |
|------|---------|--------|------|
| 绝对禁忌 | 硬排除 | **硬排除（不变）** | 安全底线 |
| 过敏冲突 | 硬排除 | **硬排除（不变）** | 安全底线 |
| 妊娠X级 | 硬排除 | **硬排除（不变）** | 安全底线 |
| MAOI+SSRI 致命交互 | 硬排除 | **硬排除（不变）** | 致命风险 |
| 无精确适应症匹配 | 评分归零 → 排除 | 标记 "off_label" + 降排序 | 医生可凭经验使用 |
| 安全数据未验证 | 标记 + 排末尾 | 标记 + 保留在候选列表 | 同上 |
| 相对禁忌 | 软警告 | 软警告 + 降排序分 0.85× | 减少推荐力度 |
| 证据等级 off_label | 排除 | 标记 "备选方案" | 新增适应症扩展 |
| 儿科禁忌 | 硬排除 | **硬排除（不变）** | 安全底线 |
| 草药补充剂治感染 | 硬排除 | **硬排除（不变）** | 无临床证据 |

### 排序优先级调整

```
最高优先: on_label + 无安全警告 + 高模型分
    ↓
中优先:  on_label + 有相对禁忌警告
    ↓
低优先:  off_label / 数据未验证 → 置于末尾但保留可见
    ↓
排除:    绝对禁忌 / 过敏 → 不展示
```

### 前端展示要求

每个推荐药物需展示：
- `safetyLevel`: 绿/黄/橙/红
- `evidenceLabel`: "一线推荐" / "备选方案" / "超说明书用药"
- `doctorReviewRequired`: true/false
- `exclusionReason`: 如被排除，说明原因

## 核心组件四：审核反馈闭环

### 审核流程

```
系统推荐(2-4个) → 医生审核 →
  ✅ 确认 → 正面信号（权重 +0.1）
  ✏️ 修改 → 记录医生选择的药物
  ❌ 拒绝 → 负面信号（权重 -0.1）
```

### 数据模型

```sql
CREATE TABLE review_log (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    recommendation_id VARCHAR(32),
    patient_id BIGINT,
    disease_cn VARCHAR(100),
    disease_standardized VARCHAR(200),
    routing_path TEXT,  -- L1→L2→L3→L4 路由记录
    system_drugs JSON,  -- 系统推荐的药品列表
    doctor_decision ENUM('confirm', 'modify', 'reject'),
    doctor_selected_drug VARCHAR(200),  -- 医生修改后选择的药
    doctor_reason VARCHAR(500),
    created_at TIMESTAMP
);
```

### 反馈利用策略

- **不自动重新训练模型**：避免引入不稳定性
- **每月生成审核周报**：
  - 高拒绝率疾病 TOP 10 → 人工检查路由规则
  - 高修改率药品类别 → 人工检查适应症匹配
  - 新增疾病模式 → 补充到 COLLOQUIAL_TO_STANDARD
- **阈值触发热修复**：单个疾病拒绝率 > 50% → 自动禁用该路由，降级为症状级推荐

## 实施计划概要

### Phase 1：知识路由核心（3-4 周）
1. 构建 DISEASE_CATEGORIES 表（204 种疾病 × 身体系统 × 病因）
2. 构建 INDICATION_TO_ATC 映射表
3. 实现四级路由引擎（KnowledgeRouter 类）
4. 替换当前 disease_mapper 的代理词逻辑
5. 回归测试 204 种疾病 → 验证路由正确性

### Phase 2：患者口语增强（1-2 周）
1. 构建 COLLOQUIAL_TO_STANDARD 口语映射表
2. 实现三层 Fallback 逻辑
3. 构建症状组合推理表
4. 测试 50 种口语输入 → 验证覆盖

### Phase 3：安全策略调整（1 周）
1. 将 SafetyFilter 中非绝对禁忌的排除规则改为标记
2. 实现排序优先级逻辑（on_label > off_label > unverified）
3. 更新前端展示组件（safety level 标签）

### Phase 4：审核闭环（1-2 周）
1. 创建 review_log 表和 API
2. 实现审核流程的前端界面
3. 实现审核周报生成脚本
4. 端到端测试

## 风险与缓解

| 风险 | 概率 | 缓解措施 |
|------|------|---------|
| 知识路由表覆盖不全 | 中 | 三层 Fallback 兜底，症状级推荐保底 |
| 医生审核工作量大 | 中 | 系统预筛至 2-4 个候选，默认选中最佳 |
| 旧系统兼容性问题 | 低 | 渐进替换：先并行运行新旧路由，逐步切换 |
| 口语表达多样性 | 高 | 持续人工扩充 COLLOQUIAL_TO_STANDARD，医生反馈驱动 |

## 成功指标

- **推荐覆盖率**：204 种疾病中，能输出推荐的比例从当前 ~95% → 100%（含降级推荐）
- **临床适用率**：DeepSeek 验证 APPROPRIATE 率从 61.1% → ≥ 80%
- **不适用率**：INAPPROPRIATE 率从 1.4% → < 1%
- **医生审核通过率**：上线 1 个月后 ≥ 70%（医生无需修改直接确认的比例）
